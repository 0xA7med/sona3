## الثغرات التقنية المحددة

### 1. RLS Infinite Recursion (`fix_rls.sql`)
**المشكلة**: بعض سياسات RLS تسبب استدعاءً متكرراً لا نهائياً → HTTP 500.  
**الموقع**: `fix_rls.sql`  
**حالة مؤقتة**: موجود حل عبر `get_my_role()`  
**الأولوية**: 🚨 عالية — يجب اختبار الحل

### 2. بيانات حقيقية في المستودع (`data_import.sql`)
**المشكلة**: الملف يحتوي أسماء كاملة، أرقام هواتف، وعناوين لأسر حقيقية.  
**المخاطرة**: كسر خصوصية المستفيدين.  
**الأولوية**: 🚨 عالية  
**الحل**: نقل الملف إلى `.gitignore` والاحتفاظ بنسخة مشفرة محلياً. إضافة وتيرة أخذ موافقة خطية على نشر البيانات.

### 3. `.env` في المستودع
**المشكلة**: ملف `.env` موّجود وليس ضمن `.gitignore`.  
**المخاطرة**: تسريب `SUPABASE_ANON_KEY` والتي رغم أنها Anon Key، إلا أنها قد تسمح بقراءة البيانات العامة.  
**الأولوية**: 🚨 عالية  
**الحل**: إضافة `.env` إلى `.gitignore` وتوفير `.env.example` للفريق.

### 4. عدم توحيد `supabase_init.sql` و `supabase_v2_migration.sql`
**المشكلة**: ملفان منفصلان للإصدار V1 و V2. أي بدء جديد يتطلب تشغيل V2 مباشرة.  
**الأولوية**: متوسطة  
**الحل**: دمج الملفين في ملف `schema.sql` واحد للتشغيل من الصفر.

### 5. عدم وجود اختبارات
**الأولوية**: 🚨 عالية  
**الوضع الحالي**: 0% تغطية اختبارية.  
**أهم المناطق للاختبار**:
- `distributionService.ts` — منطق التوزيع
- `priorityEngine.ts` — حساب الأولويات
- `idParser.ts` — تحليل الرقم القومي
- `authStore.ts` — تدفق المصادقة

### 6. خلط `campaign_id` في `case_locks`
**المشكلة**: `lockService.ts` يستخدم `campaign_id` في `case_locks` ولكن بعض SQL القديم لا يحتوي العمود.  
**الأولوية**: عالية  
**الحل**: تأكيد وجود `campaign_id` في جدول `case_locks` عبر كل ملفات SQL.

### 7. `sona3_alsaada_v2.html` (ملف HTML قديم)
**المشكلة**: ملف قديم (V1) موجود جنباً إلى جنب مع `sona3-app/` (V2).  
**الأولوية**: متوسطة  
**الحل**: أرشفة الملف القديم أو إزالته بعد التأكد من عدم استخدامه.
